## 핵심 원칙

### I. 단일 책임 원칙 및 파일 크기 제한

모든 함수와 컴포넌트는 엄격하게 단일 책임 원칙(SRP)을 따르며, 명확하게 정의된 하나의 역할만 수행합니다. 파일은 기본적으로 약 100줄 이하로 유지되어 가독성과 유지보수성을 최대화합니다.

**기본 요구사항:**
- 각 함수/컴포넌트는 명확하게 정의된 하나의 책임만 가져야 함
- 100줄을 초과하는 파일은 더 작은 모듈로 분해해야 함
- 여러 관심사를 결합한 모놀리식 함수 금지

**예외 조항:**
다음의 경우에는 100줄 제한을 예외적으로 허용합니다:
- **복잡한 UI 구조**: SVG 그래픽, 방대한 폼 구성(다중 입력, 조건부 필드), 탭 네비게이션 등
- **논리적 응집도**: 컴포넌트를 더 작은 파일로 분리하면 오히려 가독성이 저하되거나 논리적 흐름이 단절되는 경우
- **단, 예외 케이스에는 다음을 필수로 명시:**
  - 파일 상단에 명확한 주석으로 예외 이유 기록 (예: `// 복잡한 폼 구조로 인한 크기 초과`)
  - PR 본문에 파일이 커진 이유와 설계 결정 근거 상세 기술

**근거:** 작고 집중된 단위는 이해, 테스트, 디버그 및 재사용이 더 용이합니다. 다만 실무에서는 논리적 응집도와 가독성의 균형이 중요하므로, 예외는 명확한 근거와 함께 투명하게 관리됩니다.

---

### II. 포괄적 문서화

모든 코드 조각(함수, 컴포넌트, 복잡한 로직)은 그 역할과 의도를 설명하는 상세한 주석을 포함해야 합니다. 개발자는 코드 자체를 해석하지 않고 주석만 읽어서 전체 흐름과 작동을 이해할 수 있어야 합니다.

**요구사항:**
- 모든 공개 함수/컴포넌트는 목적과 매개변수를 설명하는 문서 문자열 주석이 필요
- 복잡한 로직 블록은 "왜"를 설명하는 인라인 주석이 필요
- **모든 주석과 문서는 한국어로 작성해야 함**

**근거:** 잘 문서화된 코드는 온보딩을 가속화하고, 의도를 보존합니다.

---

### III. 엄격한 계층화 아키텍처

단방향 의존성 흐름은 필수입니다: Page → Component → Hook → API. 어떤 계층도 단계를 건너뛰거나 아래 계층에 의존할 수 없습니다.

**요구사항:**
- 페이지는 약 100줄 이하로 유지되며, 훅과 컴포넌트만 조립
- 컴포넌트는 UI 렌더링과 사용자 상호작용에만 집중
- 훅은 모든 상태 관리 및 비즈니스 로직 포함
- API 함수는 HTTP 통신만 수행하고 타입화된 데이터 반환
- 역방향 또는 옆방향 의존성 허용 안 함

**근거:** 명확한 관심사 분리는 병렬 작업과 테스트 가능성을 보장합니다.

---

### IV. 의존성 최소화 & 전역 상태 관리

무거운 외부 UI 컴포넌트 라이브러리는 피해야 합니다. **전역 상태 관리는 Zustand만 허용**하며, 다른 상태 관리 라이브러리(Redux, Recoil, Context API 기반 전역 상태)는 금지됩니다.

**요구사항:**
- 외부 UI 컴포넌트 라이브러리 없음; 원시(primitive)를 직접 사용하여 컴포넌트 구축
- **전역 상태 관리: Zustand만 허용** (Redux, Recoil 등 금지)
- 각 스토어는 명확한 책임 단위로 분리하고 순환 의존 금지
- 폼 라이브러리 없음; `useState`로 기본 상태 관리

**근거:** Zustand는 가볍고 간단한 API로 복잡한 다중 페이지 상태 공유를 효율적으로 관리합니다. 제한된 선택으로 인지 부담을 최소화합니다.

---

### V. 타입 안전성 및 오류 처리

`any` 타입은 예외적이고 문서화된 경우를 제외하고 금지됩니다. 방어적 프로그래밍은 필수이며, 오류는 절대 조용히 삼켜서는 안 됩니다.

**요구사항:**
- `any` 타입 없음; 명시적 타입 또는 제네릭 사용
- 모든 외부 API 응답은 사용 전에 검증해야 함
- 사용자 입력은 경계에서 sanitize되고 검증되어야 함
- 오류는 포착되고, 맥락과 함께 로그되고, 사용자에게 전달되어야 함
- 오류 억제 금지(빈 `catch` 블록 금지)

**근거:** 타입 안전성은 개발 시간에 버그를 발견하고, 투명한 오류 처리는 사용자 신뢰를 구축합니다.

### VI. 개발 순서 강제 & 자동화된 테스트 및 빌드 관문

모든 기능 구현은 **[목업 데이터 기반 UI 개발 → Jest 컴포넌트 단위 테스트 → 실제 API 연동]** 순서를 엄격히 준수해야 합니다. 각 단계가 완료되고 검증된 후에만 다음 단계로 진행하며, 커밋 전에 **Jest 테스트와 Build를 모두 통과**해야 합니다.

**개발 순서 (필수):**
1. **목업 데이터 기반 UI 개발**: 실제 API 없이 목업 데이터로 컴포넌트 구현 및 UI 검증
2. **Jest 컴포넌트 단위 테스트**: 각 컴포넌트의 렌더링, 사용자 상호작용, 상태 변화 검증
3. **실제 API 연동**: 백엔드 API 통합 및 E2E 수동 테스트

**커밋 단위 정책 (필수) — v2.6.0 추가:**

커밋은 **논리적으로 완성된 단위**일 때만 수행합니다. 파일을 수정할 때마다 자동으로 커밋하는 것은 금지됩니다.

- **커밋 허용 시점**:
  - 하나의 기능(Feature)이 완성됐을 때 (예: Phase 6 기업 궁합 전체)
  - 독립적인 버그 수정이 완료됐을 때
  - 문서/설정 변경이 완결됐을 때
  - 리팩토링이 완료됐을 때

- **커밋 금지 시점**:
  - 파일 하나를 수정할 때마다 (중간 작업 상태)
  - 빌드나 테스트가 아직 통과되지 않은 상태
  - 기능이 절반만 구현된 상태

- **AI 에이전트 행동 지침**:
  - 파일 편집 후 자동 커밋 금지 — 빌드 검증만 수행
  - 사용자가 "커밋해줘" 또는 Phase 완료를 명시적으로 요청할 때만 커밋
  - 커밋 전 반드시 `npm test` + `npm run build` 통과 확인

**자동화된 테스트 및 빌드 관문 (필수):**
AI 에이전트 및 모든 개발자는 **커밋 전에 다음 두 명령을 반드시 실행**해야 합니다:
- `npm test` (Jest 테스트 실행) - **모든 테스트가 PASS 상태**
- `npm run build` (TypeScript 타입 체크, ESLint 린팅, 번들 생성) - **`✓ Compiled successfully` 확인**

**테스트 실패 또는 빌드 에러 처리:**
- 테스트 또는 빌드가 **하나라도 실패**할 경우, **커밋 프로세스 즉시 중단**
- AI 에이전트는 오류를 수정하고 재시도 (테스트 통과, 빌드 성공까지 반복)
- 테스트 코드 없는 기능 구현이 커밋되는 것을 원천 차단

**[Build Passed] 태그의 의미:**
- 단순 빌드 성공이 아닌, **Jest 테스트 통과 + Build 성공**을 동시에 증명
- 커밋 메시지: `<prefix>: <제목>\n\n<본문>\n\n[Build Passed]`
- 이 태그가 붙은 커밋은 비즈니스 로직(Jest) + 시스템 구조(Build) 모두 검증됨을 의미

**깨진 코드 및 WIP 커밋:**
- 깨진 코드는 절대 커밋 금지 (테스트 실패, 빌드 에러 있는 코드)
- WIP 커밋은 **로컬 브랜치에서만** 허용 (원격 저장소 푸시 금지)

**저장소 청결성 요구사항:**
- `.gitignore`에 다음 파일/디렉토리가 반드시 포함되어야 함:
  - 민감 정보: `.env`, `.env.local`, `.env.*.local` (모든 환경 파일)
  - 빌드 산출물: `node_modules/`, `.next/`, `.turbo/`, `dist/`, `build/`
  - OS 파일: `.DS_Store`, `Thumbs.db`, `*.swp`, `*.swo`
  - IDE 파일: `.vscode/settings.json`, `.idea/`, `*.iml`
- 빌드 결과물(`npm run build` 생성 파일)은 git에 커밋하지 않음
- 커밋 전 `git status`에서 예상치 못한 환경 파일이나 빌드 산출물이 staged 상태에 있는지 확인

**민감 정보 유출 사후 조치:**
- 실수로 민감 정보가 커밋된 경우, 단순 삭제 커밋이 아니라 git history에서 **완전히 제거** 필수
- `git filter-branch` 또는 `BFG Repo-Cleaner` 사용하여 git history에서 해당 정보 제거
- 해당 커밋 이후 모든 팀원에게 `git pull --rebase` 재동기화 요청
- git 호스팅 플랫폼(GitHub)에 개인 정보 노출 신고 및 토큰/API 키 즉시 재발급

**커밋 메시지 규격 (필수):**
- **언어**: 한국어
- **접두사**: Conventional Commits (`feat`, `fix`, `docs`, `style`, `refactor`, `chore`, `test`, `ui`)
- **형식**: `<prefix>: <제목>\n\n<본문>\n\n[Build Passed]`

**근거:** 엄격한 개발 순서는 UI 검증과 테스트 커버리지를 보장하며, 원자적이고 검증된 커밋은 깔끔한 git 히스토리와 안정성을 보장합니다. 저장소 청결성과 민감 정보 보호는 보안의 첫 번째 방어선입니다.

### VII. 논리적 단위 기반 PR 및 코드 리뷰

모든 작업은 **독립적인 논리적 단위**(예: 공통 UI 컴포넌트 세트, 특정 도메인의 훅 세트)를 기준으로 기능 브랜치, 커밋, 푸시 및 PR을 구성해야 합니다. 이는 개발 속도와 리뷰 품질의 균형을 맞추기 위한 것입니다.

**요구사항:**
- 모든 작업은 `main`으로부터 새로운 기능 브랜치에서 수행
- `main` 브랜치로의 직접 커밋 금지; PR을 통한 병합만 허용
- 브랜치명: `<prefix>/<feature-name>` (예: `feat/career-timing-page`, `feat/common-form-components`)
- 영문 소문자와 하이픈(-) 만 사용
- PR 범위: 하나의 컴포넌트 대신 **독립적인 논리적 단위**
  - 예) 공통 UI 버튼, 입력, 카드 컴포넌트 세트 (하나의 PR)
  - 예) 특정 도메인(career, company)의 훅 세트 (하나의 PR)
  - 예) 페이지 + 관련 컴포넌트 + 훅 (하나의 PR)
- AI 생성 코드에 대한 **인간의 코드 리뷰는 필수**

**근거:** 적절히 크기 조정된 PR은 리뷰 품질을 유지하면서 개발 속도를 높이며, 논리적 단위 기반 구성은 기능의 응집도를 보장하고 복원력 있는 코드베이스를 만듭니다.

### VIII. 조기 최적화 금지 & Phase 2 성능 프로파일링

컴포넌트 개발 중, 정확하고 단순하며 직접적인 구현에만 집중합니다. 성능 최적화(예: `useMemo`, `useCallback`)는 작성 시간에 금지되며, Phase 2에서 데이터 기반으로 측정하고 개선합니다.

**Phase 1 요구사항 (조기 최적화 금지):**
- 초기 구현 중에 성능 훅(`useMemo`, `useCallback` 등) 사용 금지
- 먼저 단순하고 읽기 쉬우며 유지보수할 수 있는 코드 작성
- 가설적 미래 요구사항을 위한 아키텍처 복잡성 금지

**Phase 2 성능 개선 요구사항 (필수):**
- 모든 성능 최적화는 **프로파일링 도구**(React Profiler, Chrome DevTools, Lighthouse 등)를 사용하여 측정
- 최적화 **전후의 수치**를 기록하고 객관적인 근거를 남김
  - 렌더링 시간, 컴포넌트 리렌더링 횟수, 번들 크기, Core Web Vitals 등
- 측정 결과는 PR 본문에 명시하여 검증 가능하게 함
- 측정 없는 성능 최적화는 금지

**근거:** 조기 최적화는 코드 명확성을 흐리고 버그를 도입합니다. 기능을 먼저 배포하고, 실제 성능 데이터를 기반으로 근거 있는 최적화를 수행합니다.

### IX. 수동 테스팅 및 사용자 검증

빌드 통과는 필수이지만 충분하지 않습니다. 모든 기능은 실제 브라우저 환경에서 수동 테스트를 거쳐야 합니다. Phase 1에서는 자동화 테스트 없이 수동 검증으로 품질을 보증합니다.

**요구사항:**
- 모든 페이지 기능 구현 후 개발 서버(`npm run dev`)에서 수동 테스트 필수
- 각 페이지별 체크리스트 작성 (spec.md의 Acceptance Scenarios 참조)
- 테스트 대상: 정상 흐름 + 에러 케이스 + 엣지 케이스
- 모바일/태블릿 환경에서도 검증
- 네트워크 지연(개발자도구 Throttling)에서도 동작 확인
- PR 병합 전에 최소 1인 이상의 인간이 기능 사용 검증 필수

**테스트 영역별 책임:**
- **Form Input**: 빈 값, 형식 오류, 유효한 값 모두 테스트
- **API 호출**: 성공, 타임아웃, 네트워크 오류 시나리오
- **State Management**: 페이지 이동 후 상태 유지, 새로고침 후 복원
- **UI Rendering**: 로딩 중, 결과 표시, 에러 상태 모든 상태 확인

**근거:** 타입 검증과 린팅은 구문 오류를 방지하지만, 실제 사용자 경험 오류를 발견하려면 수동 검증이 필수입니다.

### X. 보안 기본값 & 민감 정보 보호

모든 사용자 입력과 외부 데이터는 신뢰할 수 없습니다. 보안 원칙은 개발 시점에 내재화되어야 하며, 민감 정보 노출은 절대 허용되지 않습니다.

**입력 및 데이터 보안 요구사항:**
- **입력 검증**: 모든 사용자 입력은 형식, 길이, 범위 검증
- **XSS 방지**: HTML/스크립트 자동 이스케이프 처리 (특히 동적 콘텐츠)
- **API 통신**: HTTPS 필수, CORS 설정 확인
- **민감 정보 클라이언트 보호**: 로컬스토리지에 패스워드/토큰 절대 저장, 클라이언트 상태에서 민감 개인정보 평문 노출 금지
- **에러 메시지**: 내부 구조 노출 금지 (스택 트레이스 사용자 표시 금지)

**토큰 및 인증 정보 관리 (필수):**
- 인증 토큰은 **HttpOnly Cookie**를 사용한 안전한 저장소에만 보관 (클라이언트 상태 보관 금지)
- 로그아웃 시 토큰 완전히 삭제
- 토큰 만료 시 자동 로그아웃

**민감 정보 유출 방지 (Git & 환경 파일):**
- API 키, DB 접속 정보, 소셜 로그인 시크릿 등 민감 정보는 절대 코드에 포함하지 않음
- 모든 민감 정보는 `.env` 파일에 저장하고, `.gitignore`를 통해 git 제외 (필수)
- `.env` 파일 패턴: `.env`, `.env.local`, `.env.*.local` 등 (모든 환경 파일)
- 코드 리뷰 시 환경 파일이 `.gitignore`에 있는지, 하드코딩된 시크릿이 없는지 수시로 점검
- **실수로 민감 정보가 커밋된 경우** (필수 조치):
  - 단순 삭제 커밋이 아니라 git history에서 **완전히 제거**
  - `git filter-branch` 또는 `BFG Repo-Cleaner` 사용하여 해당 정보를 모든 커밋에서 제거
  - 팀원에게 `git pull --rebase` 재동기화 공지
  - 노출된 토큰/API 키 즉시 무효화 및 재발급

**근거:** 보안은 사후 대응보다 사전 방지가 훨씬 효과적입니다. 민감 정보 유출은 복구 불가능한 손상을 초래하며, 조기에 검출하고 완전히 제거하는 것이 필수입니다.

### XI. 성능 모니터링

조기 최적화는 금지하지만, 배포 후 성능 추적은 필수입니다. Phase 1에서는 성능 기준을 설정하고 모니터링합니다.

**요구사항:**
- **API 응답 시간 기준** (spec.md 참조):
  - CareerTiming: 3-5초
  - Consultation: 15-20초 (타임아웃 20초)
  - Compatibility: 5-8초
  - Feedback: 1-2초
- **UI 반응성**: 탭 전환 200ms 이내, 저장된 결과 재현 100ms 이내
- **번들 크기**: 초기 번들 < 200KB (gzip), 총 자산 < 1MB (이미지 제외)
- **로딩 메트릭**: Largest Contentful Paint < 3초 (3G), First Input Delay < 100ms
- **모니터링**: 배포 후 사용자 성능 데이터 수집 (Google Analytics 등)

**Phase 2에서의 최적화:**
- Phase 1 배포 후 실제 성능 데이터 분석
- 병목 지점 식별 후 우선순위 결정
- 측정 기반 최적화 수행 (useMemo, useCallback, 번들 분할 등)

**근거:** 측정되지 않은 성능 최적화는 낭비이고, 실제 사용자 환경의 데이터가 최적화 의사결정을 안내합니다.

### XII. 엔지니어링 표준화 & 일관성

모든 페이지와 컴포넌트는 공통된 디자인 시스템을 따르며, 사용자 수요 증가에 대비하여 SEO와 메타데이터를 체계적으로 관리합니다.

**Tailwind CSS 테마 표준화:**
- `tailwind.config.ts`에서 공통 테마(색상, 간격, 타이포그래피, 그림자 등)를 정의하고 준수
- 모든 컴포넌트는 하드코딩된 색상이 아닌 테마 값 사용
- 테마 변경 시 전체 애플리케이션의 UI 일관성 유지 가능하도록 구성

**페이지 메타데이터 관리 (필수):**
- 모든 페이지는 다음 메타데이터를 필수적으로 설정해야 함:
  - **SEO**: `<title>`, `<meta name="description">`, `<meta name="keywords">`
  - **Open Graph**: `<meta property="og:title">`, `<meta property="og:description">`, `<meta property="og:image">`, `<meta property="og:url">`
  - **Twitter Card**: `<meta name="twitter:card">`, `<meta name="twitter:title">`, `<meta name="twitter:description">`
- 메타데이터는 Next.js의 메타데이터 API(`metadata` 객체) 또는 `head` 태그를 통해 설정
- 각 페이지별로 고유한 메타데이터 작성 (복사-붙여넣기 금지)
- 이미지 에셋: OG 이미지는 1200×630px 이상의 고품질 이미지 사용

**근거:** 일관된 디자인은 사용자 신뢰를 높이고, 체계적인 SEO와 메타데이터 관리는 검색 엔진 최적화 및 소셜 미디어 공유 시 첫 인상을 결정합니다.

5. **Principle V** - `any` 타입 없음, API 응답 검증, 오류 억제 없음, 에러 메시지 사용자 친화적

8. **Principle VIII** - `useMemo`, `useCallback` 등 성능 최적화 미사용, Phase 2 최적화는 프로파일링 기반

12. **Principle XII** - Tailwind 테마 사용, 페이지 메타데이터(SEO, OG, Twitter Card) 설정 완료

- PR 병합 전 필수 준수

**예외 처리:**
원칙으로부터의 예외는 매우 드물어야 하며, 다음 조건에서만 허용됩니다:
- 명시적인 근거 문서화 (PR 본문에 `[Exception: <원칙명>]`)
- 리뷰어 승인 (최소 1명 이상)
- 향후 개선 계획 수립 및 명시
- 예외 내용 `EXCEPTIONS.md`에 기록

### 개정 절차

1. **제안**: 개정은 다음과 함께 문서화
   - 이전 원칙 (변경되는 내용)
   - 새로운 원칙 (제안 내용)
   - 근거 (왜 필요한가)
   - 영향 평가 (어떤 파일/프로세스 영향)
   - 버전 영향도 (MAJOR/MINOR/PATCH)

2. **검토**: 팀 논의와 합의 필수
   - 제안자가 충분한 근거 제시
   - 최소 1명 이상의 리뷰어 동의

3. **버전 관리**: 의미론적 버전 규칙 준수
   - **MAJOR**: 하위 호환성 없는 원칙 제거/재정의 (예: Zustand 사용 중단)
   - **MINOR**: 새로운 원칙 추가 또는 중대한 확장 (예: 보안 원칙 추가)
   - **PATCH**: 명확화, 표현 개선, 의미론적 변화 없음

4. **기록**: 모든 개정은 파일 최상단의 동기화 영향 보고서에 기록
